🔑 Authentication

Đăng ký, đăng nhập và lấy thông tin user. Dùng để lấy JWT token cho các API test khác.

POST /api/auth/register Đăng ký tài khoản
POST /api/auth/login Đăng nhập (admin/123)
GET /api/auth/me Lấy thông tin user từ token

🖊️ HTML Injection (Ch.2)

Chèn HTML độc hại vào các trường dữ liệu. Server có dùng stripHtml / escapeHtml để bảo vệ.

POST /api/products — HTML Injection trong name
💉 Payload: <h1>INJECTED</h1><img src=x onerror=alert('XSS')>

⚠️ Cross-Site Scripting (XSS) (Ch.3)

Các dạng tấn công XSS: Reflected, Stored, DOM-based.

POST /api/products — Stored XSS
💉 Stored XSS: Dữ liệu độc hại lưu vào database, hiển thị khi người khác xem.
GET /api/products — Reflected XSS (query params)
💉 Reflected XSS: Payload trong URL, server反射 về response.

🗄️ SQL Injection (Ch.4)

Server dùng JSON file (NoSQL), nhưng có thể test các kỹ thuật bypass login, injection qua API.

POST /api/auth/login — Bypass Login
💡 Test các payload bypass login phổ biến (dù server không dùng SQL, nhưng test cách xử lý input).
POST /api/auth/login — NoSQL Injection ($ne)
💉 NoSQL Injection: Dùng toán tử $ne để bypass nếu server dùng MongoDB.

🔓 Authentication Vulnerabilities (Ch.5)

Weak password, Brute force, Credential stuffing. Server có rate limiting (10 requests/15min cho /api/auth/login).

🔨 Brute Force Attack
⚡ Server giới hạn 10 requests/15 phút cho /api/auth/login. Hãy test rate limiting.
🧂 Credential Stuffing
💡 Dùng danh sách email/mật khẩu bị rò rỉ để đăng nhập.
🔐 Weak Password Policy
💡 Server yêu cầu password 4-128 ký tự. Test các password yếu.

🍪 Session & Cookie Security (Ch.6)

Session hijacking, Session fixation. JWT token bị đánh cắp có thể dùng để mạo danh.

🕵️ Session Hijacking
💉 Dùng token của người khác (nếu đánh cắp được) để truy cập tài khoản.
🔒 Token Expiration
💡 JWT token hết hạn sau 1 ngày. Test xem token hết hạn có dùng được không?

🌐 Cross-Site Request Forgery (CSRF) (Ch.8)

GET-based CSRF, POST-based CSRF. Server dùng JWT token trong header (không dùng cookie), nên CSRF ít hiệu quả hơn.

GET GET-based CSRF — Xóa sản phẩm
💉 Nếu API dùng GET để xóa, attacker có thể tạo link <img> để trigger.
POST POST-based CSRF — Tạo sản phẩm
💡 Server kiểm tra Authorization header. POST CSRF chỉ thành công nếu có token hợp lệ.

📁 File Upload Security (Ch.9)

Upload webshell, MIME spoofing, Extension bypass.

📤 Upload File
💉 Test upload file độc hại (webshell PHP, MIME spoofing). Server hiện chưa có endpoint upload file.

⚠️ Server hiện không có endpoint upload file. Đây là lỗ hổng bảo mật cần được thêm vào.

📂 Path Traversal & Command Injection (Ch.10)

Path traversal đọc file ngoài thư mục cho phép, OS command injection chạy lệnh hệ thống.

📂 Path Traversal
💉 Payload: ../../../etc/passwd, ..\..\..\windows\system32\drivers\etc\hosts
⌨️ OS Command Injection
💉 Payload: ; ls -la /, | dir, & echo hacked

🔐 API & JWT Security (Ch.12)

Missing Authentication, BOLA (Broken Object Level Authorization), Bypass auth, Brute force secret key.

GET /api/users/all — Missing Auth Test
💉 Endpoint này yêu cầu adminAuth. Test gọi mà không có token.
GET /api/orders/all — BOLA (Leak all orders)
💉 BOLA: User thường không được xem đơn hàng của người khác. Test với user token.
🔑 JWT Secret Key Brute Force
💡 Nếu JWT secret yếu, attacker có thể brute force để ký token giả mạo.
🔄 JWT Token Forgery
💡 Test "none" algorithm attack: Server có chấp nhận JWT với alg: "none" không?
🔍 JWT Secret Weakness
💡 Server dùng crypto.randomBytes(64).toString('hex') nếu không có env JWT_SECRET — rất mạnh. Nhưng nếu env dùng secret yếu như "secret" hay "123", có thể brute force.

Kiểm tra file .env hoặc server/middleware/auth.js:8 để xem JWT_SECRET có được cấu hình đúng không.
